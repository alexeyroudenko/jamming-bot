## MVP-дизайн

### CLI-скрипт
`analyze/embed_tags.py`

Минимальный сценарий:
- забирает все теги батчами через `GET /api/tags/get/`
- нормализует список: убирает пустые, дедуплицирует, сохраняет `count`
- отправляет слова батчами в `POST /api/tags/embeddings/`
- собирает 3D-координаты и сохраняет артефакты в `analyze/out/embed_tags/`

Пример запуска:
```bash
python analyze/embed_tags.py --projection api3d
```

### Выходные артефакты
- `tags_raw.json` — сырой нормализованный список тегов с `name` и `count`
- `tags_embeddings.json` — batched-ответы embeddings API плюс итоговые точки
- `tags_points.csv` — плоская таблица для анализа/импорта
- `tags_points.ply` — облако точек для 3D-инструментов
- `metadata.json` — источник данных, режим проекции, схема `PLY` и пути к артефактам

### PLY
Для MVP достаточно ASCII `PLY` с вершинами и минимальным набором полей:
- `x`
- `y`
- `z`
- `count`
- `label_index`

`label` как строку внутрь `PLY` не класть: для совместимости оставить строковые имена в `metadata.json` как `labels_by_id`, где индекс совпадает с `label_index`.

### Fallback-режим
Основной режим: `--projection api3d`, то есть `vectors3d_current`.

Если базовая 3D-развёртка упирается в лимиты или даёт слабую сцену:
- `--projection api3d_alt` — использовать `vectors3d_alt` из того же API как альтернативный базис без нового ML pipeline
- `--projection api2d_z` — брать `vectors2d_current` и достраивать `z` детерминированно, чтобы pipeline не ломался даже при недостаточном качестве `vectors3d`

> [!warning]
> У embeddings API есть лимит `max_words`, поэтому полный экспорт надо делать батчами. Для MVP это решается на стороне `analyze/embed_tags.py`, без изменения runtime-сервисов.
